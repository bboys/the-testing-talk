# What is testing?
## What usually happens - Write some code - Run your program with some different parameters and/or matrices - Run it in parallel for one test problem? - Run it with valgrind for one test problem?
## Now - Does it work for any combination of parameters/any matrix? - Does it work with any number of processors?
## Idea behind automated testing - Once your program works, it should never break for previous test cases

## Different types of tests - Unit tests - Integration tests - Acceptance tests
### Unit Tests - Test a small part of the program - Low level
#### Example: `backslash(A, b)`, with `A` a sparse spd matrix, should call `cg(A, b)`.
### Integration Tests - Test coupled systems
#### Example: - `backslash(A, b)` should return `x`.
### Acceptance Tests - Test user requirements
#### Example: - Solver should be able to run parallel on 16 cores

# Other good practices
## Use version control - Even when working alone - Use sane commit messages
## Write readable code - Even when working alone - Use one coding style - Write it so someone else can read and run your code the next day - This includes writing documentation
## Script everything - Never manually run experiments - Improves reproducibility - Reduces the need to store results
## Use libraries - If something already exists, don't write it yourself
